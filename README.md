# EnvyAgent

EnvyAgent is a FastAPI-based multi-agent equity research service. It accepts natural-language stock questions, resolves ticker context, pulls market and financial data from Financial Modeling Prep (FMP), runs valuation and growth/risk computations, and returns a narrative response generated by OpenAI.

This repository ships both API and UI:

- `main.py` - backend orchestration and all agent layers
- `chat.html` - lightweight browser chat client served by FastAPI
- `streamlit_app.py` - Streamlit chat client for local or hosted use

---

## What EnvyAgent Actually Does

EnvyAgent is not a single prompt wrapper. It is a staged analysis pipeline with specialized responsibilities:

1. Query understanding (intent + ticker extraction)
2. Context resolution (single/multi ticker and follow-up handling)
3. Data collection from FMP endpoints
4. Valuation and metrics computation
5. Route selection (news-only, financials-only, full/comparison)
6. Response generation with disciplined prompt templates
7. Session memory updates for follow-up continuity

The result is a stateful stock-analysis assistant that can handle:

- single ticker analysis (`AAPL`)
- multi-ticker comparison (`NVDA vs AMD`)
- follow-up turns without repeating ticker (`What about margins?`)
- route specialization (`latest news on TSLA`, `show financials for MSFT`)

## Architecture Overview

### 1) API Layer (FastAPI)

The API surface is intentionally small:

- `POST /query` - primary conversational endpoint; uses multi-agent orchestration and session memory
- `GET /analyze` - returns structured pipeline output for a single ticker (JSON)
- `POST /chat` - direct OpenAI pass-through (no stock pipeline)
- `GET /` and `GET /chat.html` - serves the built-in chat UI

`/query` is where the multi-agent architecture lives.

### 2) Conversation/Memory Layer

Session state is stored in-process in `session_memory` and keyed by:

1. `session_id` from request body (preferred)
2. `X-Session-Id` header fallback
3. client host fallback
4. `"anonymous"` fallback

Each session stores:

- `current_ticker` - last resolved ticker for follow-ups
- bounded `messages` history - latest user/assistant turns

This lets EnvyAgent resolve follow-up questions that omit ticker symbols.

### 3) Understanding + Routing Layer

EnvyAgent extracts:

- `tickers` (possibly multiple)
- `primary_intent` in fixed taxonomy
- `secondary_intents` for blended responses
- `is_follow_up` signal

Allowed primary taxonomy:

- `valuation`
- `risk`
- `growth`
- `comparison`
- `full_analysis`

Routing logic then chooses one of several response paths:

- clarification path (if no ticker can be resolved)
- comparison path (if comparison intent / multiple tickers)
- news-only path
- financials-only path
- full structured analysis path

### 4) Data Agent Layer

`data_agent(symbol)` pulls and normalizes data from FMP:

- company profile
- annual income statements
- latest income/balance/cash-flow snapshots
- dividend history
- historical prices (2017 through configured max year)
- trailing-twelve-month EPS from latest 4 quarters

This creates one normalized raw payload consumed by downstream agents.

### 5) Valuation Agent Layer

`valuation_agent(raw_data)` computes equity signals and rejects low-confidence cases.

Key computations include:

- intrinsic value proxy from historical average P/E and EPS
- undervaluation percent vs current price
- quality and balance-sheet ratios (ROE, gross margin, P/S, P/B, FCF ratio, D/E)
- growth indicators (EPS CAGR, PEG)
- annualized return projection
- dividend yield

Important behavior: the valuation agent can return `None` when data quality or growth thresholds are not met, and orchestration handles that gracefully.

### 6) Prompting/Response Agent Layer

EnvyAgent builds structured prompts around:

- normalized metric JSON
- ticker-specific news context
- primary/secondary intent guidance
- "disciplined value-investor" answer style

OpenAI is called through `call_openai()` using the Responses API with:

- model default from `OPENAI_MODEL_DEFAULT`
- `store: false` to avoid server-side storage

## End-to-End `/query` Flow

For each user request:

1. Store user message in session memory.
2. Parse intent/tickers (`understand_query()`).
3. Resolve current context and fallback ticker memory (`resolve_query_context()`).
4. Handle special branches:
   - missing ticker clarification
   - insufficient comparison ticker count
   - news-only route
   - financials-only route
5. Build per-ticker analysis context (`build_multi_ticker_context()`).
6. Generate final response from structured inputs (`generate_response_from_structured_input()` or route-specific generator).
7. Store assistant response and return `ChatResponse`.

The orchestration entry point is `orchestrate_query_response()`.

## API Reference

### `POST /query`

Primary natural-language endpoint.

Request body:

```json
{
  "query": "Compare NVDA vs AMD on valuation and risk",
  "session_id": "optional-client-session-id"
}
```

Response:

```json
{
  "response": "Narrative answer text...",
  "model": "gpt-4.1-mini"
}
```

Behavior notes:

- supports single ticker, multi ticker, and follow-up turns
- resolves company names to tickers when possible
- can return clarification-style responses when ticker context is missing

### `GET /analyze`

Returns structured JSON for one ticker:

`/analyze?ticker=AAPL&query=optional+intent+hint`

Includes:

- normalized valuation output
- chart/plot data
- intent metadata
- compact news context
- generated summary object

### `POST /chat`

Direct OpenAI endpoint (no stock analysis pipeline).

Request:

```json
{
  "prompt": "Explain discounted cash flow in plain language.",
  "model": "gpt-4.1-mini"
}
```

Response:

```json
{
  "response": "Model output...",
  "model": "gpt-4.1-mini"
}
```

## UI Clients

### `chat.html` (served by FastAPI)

- available at `/` and `/chat.html`
- sends `POST /query` to same-origin API by default
- persists browser session id in localStorage (`ENVY_SESSION_ID`)

If opened as a local file (`file://`), set:

```js
localStorage.setItem("ENVY_API_BASE_URL", "http://127.0.0.1:8000");
```

### `streamlit_app.py`

Alternative chat UI that targets `/query`:

- reads API base from Streamlit secrets or `ENVY_API_URL`
- supports runtime override in UI
- includes hosted-vs-local connection hints

## Configuration + Operational Notes

- default model: `OPENAI_MODEL_DEFAULT = "gpt-4.1-mini"`
- OpenAI calls use `store: false`
- intent taxonomy is fixed to five categories
- memory is in-process only (non-persistent across restarts)
- multi-ticker processing is capped (`MAX_MULTI_TICKERS = 5`)

## Security and Hardening

Current code has one production-critical issue:

- FMP API key is hardcoded in `main.py` as `API_KEY`

Recommended immediate fixes:

1. Move FMP key to environment variable (e.g. `FMP_API_KEY`).
2. Rotate/revoke any key that has been committed.
3. Add `.env`/secret-management workflow for local and Render.

Optional hardening improvements:

- add request-level logging/trace IDs around orchestration stages
- add retry + backoff policy for external API calls
- replace in-memory session storage with Redis for multi-instance deployment
